Pathways17 / 22

Pathways – Définition

  • Pathways est l'infrastructure développée par Google pour exécuter PaLM à très grande échelle, servant de système d'orchestration distribué pour les calculs de ML sur des milliers d'accélérateurs.
  • Elle permet de coordonner des milliers de TPU/GPU pour entraîner un modèle colossal réparti sur de nombreux nœuds.
  • Objectif principal : maximiser l'efficacité matérielle et réduire les temps d'inactivité lors de l'entraînement et de l'inférence.

Idée centrale

Pathways agit comme une couche de coordination à grande échelle : elle distribue les tâches, active les ressources au bon moment et maintient un haut niveau d'utilisation du matériel.

Pathways18 / 22

Pathways – Activation partielle & Multitask Learning

  • Activation partielle des paramètres : seules les parties du modèle nécessaires à la tâche sont activées, ce qui réduit l'utilisation mémoire et le temps de calcul.
  • Multitask Learning (MTL) : un modèle unique peut apprendre plusieurs tâches simultanément, en partageant des paramètres communs.
  • Pathways aide à réduire le catastrophic forgetting en permettant l'entraînement de plusieurs tâches sans effacer les performances antérieures.
L_MTL = Σ(t=1→T) λ_t L_t(θ)

Variables

  • T : nombre de tâches
  • λ_t : poids relatif de la tâche t
  • L_t(θ) : loss de la tâche t
  • θ : paramètres partagés du modèle
Pathways19 / 22

Pathways – Orchestration & parallélisme

  • Orchestration distribuée : Pathways coordonne les TPU/GPU et lance les opérations dès que les données sont prêtes, même avec des dépendances complexes.
  • Parallélisme de données : différents lots de données sont traités simultanément sur plusieurs accélérateurs.
  • Parallélisme de modèle : différentes parties du modèle sont distribuées sur plusieurs machines pour entraîner des modèles trop grands pour une seule puce.

Avantages clés

  • Permet un apprentissage multi-tâches massif
  • Optimisation de l'utilisation des ressources, avec une utilisation proche de 100 % des TPU
  • Réduction des coûts et de l'énergie par rapport aux anciens paradigmes distribués
Entraînement20 / 22

Entraînement du modèle

Dataset & objectif

  • PaLM est pré-entraîné sur un corpus massif de ~780 milliards de tokens : pages web filtrées, livres, Wikipedia, presse, code source et conversations.
  • Ce corpus reprend en partie les mêmes sources que LaMDA, avec une forte diversité linguistique et contextuelle.
  • Tâche principale : next token prediction, de façon autoregressive.
  • Le modèle apprend à maximiser la probabilité du prochain token à partir du contexte précédent.
L = - Σ(i=1→N) Σ(j=1→|V|) y_i,j log(ŷ_i,j)

Optimisation & scaling

  • La cross-entropy loss compare la distribution prédite à la distribution réelle one-hot.
  • Adafactor réduit l'utilisation mémoire en factorisant les moments nécessaires à l'optimisation.
  • Batchs très larges : milliers de tokens traités simultanément.
  • Data Parallelism : plusieurs lots sont traités en parallèle.
  • Model Parallelism : différentes parties du modèle sont réparties sur plusieurs TPU/GPU.
  • Objectif final : maximiser l'utilisation matérielle, avec une MFU très élevée.
Dataset : ~780B · Objectif : next token · Loss : cross-entropy · Optimizer : Adafactor · Scaling : data + model parallelism
Performance21 / 22

Capacités & performances

Benchmarks & raisonnement

  • Benchmarks utilisés : BIG-Bench, MMLU, HumanEval et autres suites de compréhension, génération et raisonnement.
  • Ils permettent de comparer PaLM à d'autres grands modèles comme GPT-3, Gopher ou Chinchilla.
  • Les tests few-shot évaluent la capacité à s'adapter avec quelques exemples, sans entraînement supplémentaire.
  • Les performances sont mesurées en compréhension, génération, raisonnement et code.

Scaling & CoT

  • Les tâches de reasoning restent difficiles car elles exigent logique multi-étapes et maintien du contexte long.
  • Chain-of-Thought (CoT) aide le modèle à générer une pensée étape par étape avant la réponse finale.
  • CoT améliore la précision sur les problèmes de logique, de mathématiques et de raisonnement complexe.
  • L'augmentation du nombre de paramètres et de tokens améliore les performances de raisonnement.
  • PaLM 540B performe mieux sur les tâches complexes, surtout en few-shot et avec CoT.
Benchmarks : BIG-Bench, MMLU, HumanEval · Few-shot · CoT · Scaling = meilleures performances

Attention causale

Attention(Q, K, V) = softmax((Q · K + M) / d) · V

Entraînement

Entraînement Cross-Entropy Loss

Codex apprend à prédire le token suivant.

L = -(1/T) Σ log P(tᵢ | ttᵢ ; θ)
  • tᵢ = token réel attendu
  • P = probabilité prédite
  • θ = paramètres du modèle
  • T = nombre total de tokens

Apprentissage auto-supervisé : le code sert lui-même de label.

Pipeline dentraînement

  • Point de départ : GPT pré-entraîné
  • Collecte et nettoyage dun corpus de code
  • Tokenisation adaptée à la programmation
  • Fonction de coût : Cross-Entropy Loss
  • Optimiseur : AdamW ou Adam selon les implémentations décrites
  • Scheduler : cosine decay ou variante proche

Objectif : réduire lécart entre le token prédit et le token réel.

valuation

valuation HumanEval

Pourquoi pas BLEU ?

Deux programmes différents peuvent être corrects sans avoir le même texte. Une simple comparaison textuelle est donc insuffisante.

Message clé : le code doit fonctionner, pas seulement ressembler à une référence.

HumanEval

  • Benchmark proposé par OpenAI
  • 164 problèmes de programmation
  • Chaque problème contient une signature de fonction, une docstring et des tests unitaires
  • Le code généré est exécuté

Codex est donc évalué fonctionnellement : il faut passer les tests.

Métrique

Métrique pass@k

pass@k = 1 - C(n-c, k) / C(n, k)
  • n = nombre total de solutions générées
  • c = nombre de solutions correctes
  • k = nombre de solutions testées

Exemple

Si le modèle génère 100 solutions et que 28 sont correctes :

  • pass@1 28 %
  • pass@100 peut être beaucoup plus élevé

Plus k est grand, plus le modèle a de chances de proposer au moins une solution correcte.

RMSNorm

RMSNorm Normalisation

Problème

[0.002, 3200, 0.0001, 850]

Après plusieurs calculs, les valeurs peuvent devenir instables.

Solution

RMSNorm divise par la racine carrée de la moyenne des carrés.

[0.001, 1.70, 0.00, 0.43]
  • Placée avant les blocs Self-Attention et FFN
  • Aide à stabiliser lentraînement
  • Plus simple et souvent plus rapide que LayerNorm
RMS(x) = (moyenne(x²)) Sortie = x / RMS(x)
Self-Attention et RoPE

Self-Attention + RoPE

Self-Attention

  • Chaque token crée trois vecteurs : Q, K, V
  • Le modèle compare Q avec K pour estimer limportance
  • Score = Q · K / d
  • Masque causal : pas daccès aux tokens futurs
    • Attn(Q,K,V) = softmax((QK / d_k) + M)V
  • Softmax transforme les scores en probabilités
  • Résultat = somme pondérée des V

RoPE

  • RoPE = Rotary Position Embedding
  • Encode la position des tokens dans la séquence
  • Un même mot na pas exactement la même représentation selon sa position
"je" en position 1 "je" en position 5

Intuition : la position est injectée par une rotation du vecteur, ce qui aide le modèle à gérer lordre des mots.

FFN SwiGLU

FFN SwiGLU Réseau Feedforward

Après lattention, le réseau Feedforward traite chaque token individuellement.

4096 11008 SwiGLU 4096
FFN(x) = W · SwiGLU(Wx)
SwiGLU(x) = (xW) SiLU(xW) W

SwiGLU améliore souvent lexpressivité du bloc feedforward sans changer la logique générale du Transformer.
